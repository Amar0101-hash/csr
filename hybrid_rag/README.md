# hybrid_rag

The **Hybrid RAG** package: the "proper" hybrid that fuses **dense vector +
sparse full-text (LanceDB) + Neo4j graph expansion** with multi-signal
Reciprocal Rank Fusion, rewarding **consensus** so a source found by more than one
signal outranks any single one.

## Why it's more than vector + graph concatenated

Every signal contributes on the **same rank-based (RRF) scale**, so a source that
is BOTH semantically retrieved AND linked in the clinical-entity graph is scored
above either signal alone. That cross-document agreement is exactly the
consistency a CSR needs — an endpoint *defined* in the SAP, *described* in the
Protocol, and *measured* in a TFL table — and it's what a pure vector or pure
graph retriever each miss on their own.

## The three signals

1. **Dense vector** (LanceDB) — semantic match over Titan embeddings.
2. **Sparse full-text** (LanceDB) — exact clinical terms / codes.
3. **Graph expansion (Neo4j)** — starting from the top vector+FTS seeds, traverse
   the clinical-entity graph:

   ```cypher
   (seed:Chunk)-[:MENTIONS]->(e:Entity)<-[:MENTIONS]-(other:Chunk)
   ```

   Each shared entity is weighted by **rarity** — `1 / (1 + entity_frequency)` —
   so specific clinical terms count for more than ubiquitous ones. Chunk ids are
   content hashes shared by LanceDB and Neo4j, so seeds map straight across with
   no id translation.

Scoring: `score(c) = w_vector·rrf(vrank) + w_fts·rrf(frank) + w_graph·rrf(grank)`
with default weights `1.0 / 1.0 / 0.5` (graph is a softer, *expanding* signal).
Results also carry a **provenance** string (`"graph+vector"`, `"fts+graph+vector"`,
…); the `+` count is the consensus signal surfaced in the Compare UI.

It also keeps `vector_rag`'s **guaranteed-table** reservation for TFL result
tables, and **degrades gracefully**: if Neo4j is unreachable or the entity graph
is missing, the graph signal contributes nothing and the hybrid falls back to
vector + FTS instead of erroring.

```
hybrid_rag/

```

## Usage

```python
from vector_rag.config import Settings
from hybrid_rag import build_hybrid_retriever

hr = build_hybrid_retriever(Settings())          # opens LanceDB + a Neo4j driver
hits = hr.retrieve("primary effectiveness endpoint results", k=10)
for h in hits:
    print(h.score, h.chunk.doc, h.provenance, h.chunk.section_path)
hr.close()
```

`retrieve()` returns `list[RetrievedChunk]` (the same type `vector_rag` and
`graph_rag` produce), so the hybrid drops straight into the shared generation
core (`graph_rag.generate.author_from_retrieved`). In the app, `backend` selects
it when the generation method is `hybrid`.

## Dependencies

- `vector_rag` — VectorStore (LanceDB), TitanEmbedder, models, `RetrievedChunk`.
- `graph_rag.gr_config` — Neo4j connection settings (host/creds/database).
- A running Neo4j containing the `(:Chunk)-[:MENTIONS]->(:Entity)` entity graph.

> **Reproducibility note.** The entity graph the hybrid expands over
> (`Chunk` / `Entity` / `MENTIONS`) currently exists in Neo4j from an earlier
> build but is **not regenerated by current code** (the old builder was removed in
> the pure-vector refactor). If Neo4j is wiped, that layer must be rebuilt before
> the graph signal works again — a small entity-graph builder in `graph_rag` is
> the recommended follow-up.

## Set aside: the NetworkX path

An in-memory NetworkX entity graph exists at `vector_rag/knowledge/graph_store.py`
and an earlier version of this retriever used it. Per the current decision the
hybrid uses **Neo4j only**; the NetworkX `GraphStore` is kept (imported nowhere)
for the planned AWS design where a per-study graph is loaded into memory for the
session.
